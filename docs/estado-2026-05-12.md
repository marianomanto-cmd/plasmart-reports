## 1. Qué es

Dashboard interno de reportería de campañas digitales para Plasmart
(grupo Transfil, Córdoba). Consolida Google Ads + Meta Ads + GA4, con
análisis automático generado por Claude. Lo usan ~5-10 personas internas;
~20-50 campañas activas; refresco semanal (cron diario 06:00 ART).

## 4. Modelo de datos (Postgres)

Tablas:
- `dim_campaign` — dimensión de campañas (publisher, external_id, name, type, status)
- `fact_campaign_daily` — hechos diarios (impressions, clicks, conversions, cost_ars, revenue_ars, roas calculado)
- `fact_ga_daily` — datos diarios de GA4
- `ai_analysis_cache` — cache de Claude por hash de filtros + max_date
- `ai_analysis_log` — log permanente de análisis generados (visible en /admin)
- `ingestion_log` — log de cada corrida de la edge function

Vistas materializadas:
- `mv_campaign_weekly`, `mv_campaign_monthly`, `mv_publisher_summary`

RPCs principales:
- `dashboard_kpi_totals`, `dashboard_publisher_comparison`,
  `dashboard_daily_totals`, `dashboard_anomalies`,
  `dashboard_last_ai_call` (cooldown), `refresh_all_materialized_views`.

RLS habilitado en todas. Política de lectura: usuarios autenticados con
email `@transfil.com.ar`. Moneda única: ARS.

## 5. Frontend

App Router con tres pestañas en `/dashboard`:

1. **Resumen** (`/dashboard`) — KPIs principales con sparklines y deltas,
   evolución de costo, top de campañas, análisis IA.
2. **Comparativa** (`/dashboard/comparativa`) — tabla GAds vs Meta por
   métrica, con badges de anomalías.
3. **Detalle** (`/dashboard/detalle`) — tabla completa por campaña,
   sección GA4 (KPIs + source/medium), badges de anomalía.
4. **Corey Haines** (`/dashboard/corey-haines`) — análisis IA alternativo
   con prompt y framework distinto (lente de growth).

Filtros (en URL, persistentes): rango de fechas (selector exacto o
slider de período), publisher, tipo de campaña, búsqueda por nombre.

Panel admin (`/admin`): últimas 20 ingestas, freshness de datos,
botón "forzar ingesta ahora" (cooldown 10 min), log de análisis IA.

Auth: `/login` con wordmark + botón Google. Middleware
(`proxy.ts` + `lib/supabase/middleware.ts`) refresca sesión y bloquea
cualquier email fuera de `@transfil.com.ar`.

## 6. Pipeline de datos

1. `pg_cron` dispara `ingest-reports` (edge function) cada día a las
   09:00 UTC (06:00 ART).
2. La función:
   - Lee Sheets de Drive (GAds, Meta).
   - Llama a la API de GA4.
   - Normaliza y hace upsert en `dim_campaign`, `fact_campaign_daily`,
     `fact_ga_daily`.
   - Refresca las vistas materializadas.
   - Inserta una fila en `ingestion_log`.
3. El dashboard consulta vistas/RPCs via Supabase desde Server Components.

## 7. Motor de análisis IA

- **API routes:** `app/api/analyze/route.ts` (resumen) y
  `app/api/corey-haines/route.ts` (lente growth).
- **Prompts:** `lib/ai/prompt.ts`, `lib/ai/corey-prompt.ts`,
  `lib/ai/account-context.ts` (contexto estable de Plasmart).
- **Cache:** `ai_analysis_cache` por hash de filtros + max_date. Si el
  filtro y la fecha máxima de datos no cambiaron, se devuelve el cacheado.
- **Cooldown:** server-side, RPC `dashboard_last_ai_call`, 60 min por
  usuario y por analizador. La API route responde 429 si pasó menos.
- **Log:** cada llamada exitosa va a `ai_analysis_log` (visible en /admin
  con vista expandible).
- **UI:** botón "Regenerar análisis" + botón "Descargar PDF" del análisis.
- **Costo:** ~USD 0.02–0.05 por análisis con Sonnet 4.6.

## 10. Decisiones de diseño vigentes

- **Paleta:** se migró de la propuesta inicial (negro + cobre) a la
  industrial-azul que vino con Tremor (PR #6). Conviene revisar si se
  vuelve a la paleta original (negro/cobre) o se queda con esta.
- **Componentes:** Tremor + Recharts para todo lo nuevo; los charts
  originales en SVG vanilla quedaron en `components/charts/`.
- **Server Components por default**, Client Components solo donde hace
  falta interactividad (filtros, botones de IA, tablas con orden).
- **Idioma:** código en inglés, UI/copy/commits en español rioplatense.

## 11. Features pospuestas (no retomar sin pedido)

- **Reporte semanal por mail** (Resend + pg_cron). Prototipado en PR #8,
  revertido en PR #9. Razones: fricción operativa (verificar dominio,
  configurar envs y secrets) y ruido (análisis viejo en bandeja mientras
  se itera). Código disponible en commit `0b7d224` para retomar.

## 12. Backlog conocido

- CPA muestra "0,00" cuando son fraccionarios chicos (`<1` ARS). Bug menor.
- Tooltips nativos en nombres truncados del top de campañas (SVG, 3 líneas).
- Gráfico de evolución: líneas de GAds y Meta se pegan visualmente con
  valores cercanos.
- Mobile: usable pero no optimizado para celular (KPIs apilan, tabla con
  scroll horizontal).
- Notificaciones por email cuando falla la ingesta semanal (Fase 6.x
  opcional, sin pedido).
- Backfill GA4 vía edge function parametrizada por rango (pendiente).

## 13. Posibles temas para discutir con Claude web

Algunos frentes abiertos donde tiene sentido pensar antes de implementar:

- **Paleta de marca:** ¿se queda la azul-industrial actual o se vuelve a
  la negro+cobre original alineada a plasmartcba.com?
- **Lanzamiento (Fase 7):** comunicación interna, capacitación corta al
  equipo, handoff de operación (quién monitorea ingesta y costos de IA).
- **Reporte semanal por mail:** revisitar o descartar definitivamente.
- **Mobile-first del dashboard:** ¿vale la pena dado que el uso real es
  desktop?
- **Backlog del IA:** ¿comparar resúmenes entre semanas? ¿proponer
  acciones concretas en lugar de descripciones?
- **Modelo de Claude:** ¿probar Haiku para bajar costo 10x? ¿O dejar
  Sonnet por calidad del análisis?
- **Más fuentes de datos:** ¿LinkedIn Ads? ¿Llamadas / WhatsApp inbound
  para cerrar el loop de conversión?
